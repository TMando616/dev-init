# 要件定義: Phase 6 - 管理者と生徒の完全分離

## 概要

**目的**: 現状は単一の `users` テーブルに `role`（`admin`/`user`）列を持たせ、同一UIに管理者だけ項目を追加表示する構成になっている。これを見直し、**管理者を生徒（学習者）から完全に分離**する。管理者はカリキュラムを進める「生徒」である必要がないため、専用の `admins` テーブル・専用認証・専用UIに切り出す。

**背景・課題**:
- 管理者と生徒が同一テーブル・同一トークン・同一画面シェルに同居しており、責務と権限境界が曖昧。
- 管理者がダッシュボードや演習など生徒向け機能を保持しており、「管理者は生徒ではない」という業務実態と乖離している。
- UIがロール条件分岐（`user.role === 'admin'`）に依存し、画面の混在・取り違えリスクがある。

**対象ユーザー**:
- 生徒（学習者）: カリキュラム（学習資料・演習）を進める。管理機能は一切見えない。
- 管理者: コンテンツ（Lesson/Category/Material）と生徒アカウントを管理する。カリキュラムは進めない。

**今回の方針決定（確定事項）**:
- 管理UIは **Next.js 内で route group により完全分離**（独立アプリ化はしない）。
- 管理者アカウントは **既存管理者による招待作成**。初期管理者はシーダーで投入。**管理者の公開登録は不可**。
- 管理者の権限は **単一レベル**（管理者間の権限差は設けない。将来RBACが必要になれば拡張）。

---

## ユーザーストーリー

### 管理者向け

**US-1: 管理者専用ログインでログインする**
> As a 管理者 / I want to 生徒とは別の管理者専用ログインからログインしたい / So that 生徒と認証経路・権限が明確に分かれる。

受け入れ条件:
- [ ] 管理者は `/admin/login`（生徒の `/login` とは別）からログインできる
- [ ] 管理者ログインは `admins` テーブル・専用ガード（`admin`）で認証される
- [ ] ログイン成功時、管理者トークンが発行され管理ダッシュボード（`/admin`）に遷移する
- [ ] 生徒の認証情報（`users`）では管理者ログインに成功しない

**US-2: 管理者専用UIでコンテンツを管理する**
> As a 管理者 / I want to 生徒と完全に分離された管理者専用UIで操作したい / So that 画面の取り違えなく管理業務に集中できる。

受け入れ条件:
- [ ] 管理画面は生徒アプリと別レイアウト（別シェル・別ナビゲーション）で表示される
- [ ] 管理画面に生徒向け機能（演習実行・カリキュラム進捗）への導線が存在しない
- [ ] Lesson / Category / Material のCRUDが管理画面から行える（既存機能の移設）
- [ ] 管理者は演習の実行・提出・進捗保存を行えない（生徒専用機能として遮断される）

**US-3: 生徒アカウントを管理する**
> As a 管理者 / I want to 生徒（学習者）アカウントの一覧・作成・編集・削除を行いたい / So that 受講者を運用管理できる。

受け入れ条件:
- [ ] 管理画面から生徒（`users`）の一覧を確認できる
- [ ] 生徒アカウントの作成・編集・削除ができる
- [ ] 生徒一覧・操作対象に管理者アカウントは混在しない（`users` は純粋に生徒のみ）

**US-4: 別の管理者を招待作成する**
> As a 管理者 / I want to 管理画面から別の管理者を作成したい / So that 公開登録を許さずに管理者を安全に増やせる。

受け入れ条件:
- [ ] 管理画面から新規管理者（`admins`）を作成できる
- [ ] 管理者の公開登録エンドポイント（`/admin/register` 等の未認証作成経路）は存在しない
- [ ] 管理者作成APIは `auth:admin` で保護される
- [ ] 自分自身の削除はできない（既存の self-delete 防止を踏襲）

### 生徒向け

**US-5: 生徒は管理機能を一切意識しない**
> As a 生徒 / I want to 管理機能が見えない純粋な学習UIを使いたい / So that 自分に関係ない機能に惑わされず学習できる。

受け入れ条件:
- [ ] 生徒アプリのナビゲーションに「管理者メニュー」が表示されない
- [ ] 生徒トークンで管理系API（`auth:admin` 配下）を叩くと拒否される（401/403）
- [ ] 生徒の既存学習データ（Submission・進捗）は移行後も保持される

### システム / 移行

**US-6: 既存の管理者データを安全に移行する**
> As a 運用者 / I want to 既存 `users` 内の管理者を `admins` テーブルへ移行したい / So that 分離後も既存管理者がログインできる。

受け入れ条件:
- [ ] 既存 `admin@example.com` が `admins` テーブルへ移行される（シーダーで再現可能）
- [ ] `users` テーブルから `role` 列が削除される（マイグレーションで実施）
- [ ] `role` に依存した既存コード（`User::isAdmin()`・`AdminMiddleware`・フロントの `role` 分岐）が除去または置換される
- [ ] 移行後、生徒のレコードはすべて学習者として扱われる（管理者レコードは `users` から消える）

## データモデル要件

### `admins` テーブル（新規）
| カラム | 型 | 備考 |
|--------|----|----|
| id | bigint PK | |
| name | string | |
| email | string unique | ログインID |
| password | string | ハッシュ |
| timestamps | | |

- カリキュラム系リレーション（Submission 等）は持たない。
- `Admin` モデルは `Authenticatable` を継承し `HasApiTokens` を使用。

### `users` テーブル（変更）
- `role` 列を削除。以後 `users` は生徒専用。
- 既存のリレーション（Submission など）は維持。

### 認証ガード
- `config/auth.php` に `admin` ガード（provider=`admins`）を追加。
- Sanctum はトークンを発行元モデル（User / Admin）に紐づけるため、生徒トークンと管理者トークンを別系統として扱える。
- 管理系ルートは `auth:admin`、生徒系ルートは従来どおり `auth:sanctum`（users provider）で保護。

## 非機能要件・セキュリティ

- [ ] 管理者の公開登録経路を一切設けない（招待作成のみ）。
- [ ] 管理系APIはガードレベルで保護し、生徒トークンでの到達を不可能にする。
- [ ] フロントの管理エリアは `middleware.ts` 等でエッジ保護し、未認証・生徒トークンでのアクセスをログイン画面へ誘導する（クライアント側分岐のみに依存しない）。
- [ ] 既存テストを破壊しない。`role` 廃止に伴う既存テスト（AdminUserManagementTest 等）は新方式に合わせて更新する。
- [ ] 移行はマイグレーション＋シーダーで再現可能（`migrate:fresh --seed` が通る）。

---

## 技術的負債・クリーンアップ（本フェーズで併せて対応）

- [ ] **未使用の iframe コード実行ロジックを削除する**。`frontend/src/lib/codeRunner.ts`（`CodeRunner` クラス / `runner`）は設計当初の「軽い JS はフロント iframe で即実行」案の名残だが、現在どこからも import されていないデッドコード。実際の実行は全言語が `lessons/[id]/page.tsx` の `handleRun` から `api.post('/execute')` 経由で **バックエンドの Docker サンドボックスに一本化**されている（`expected_output` との突合をサーバー側で統一するため）。
  - 対応: `codeRunner.ts` を削除。import 参照が無いことを `grep -rn "codeRunner\|CodeRunner\|runner"` で再確認してから消す。
  - 補足: もし将来クライアント実行を復活させる場合でも、合否判定はサーバー側に残す前提（クライアント実行は結果改竄が可能なため）。

## ギャップ分析（既存コードベースとの乖離）

現状実装と本要件の差分。設計フェーズで対応方針を具体化する。

| 領域 | 現状 | 本要件での変更 |
|------|------|----------------|
| テーブル | `users.role` で管理者兼用 | `admins` 新設・`users.role` 削除 |
| 認証 | 単一 Sanctum ガード、共通 `/login`・`/register` | `admin` ガード追加・管理者専用ログイン分離 |
| 認可 | `AdminMiddleware`（`isAdmin()` チェック） | `auth:admin` ガードに置換、`AdminMiddleware`・`isAdmin()` 廃止 |
| 管理API | `/api/users`・各CRUDが `users` 認証下の `admin` ミドルウェア | `auth:admin` 配下に再編。生徒管理(`/users`)と管理者作成(`/admins`)を分離 |
| AuthController | login/register/CRUD が users 前提で混在 | 生徒用と管理者用に責務分離（Controller→Service→Repository 維持） |
| フロント シェル | 単一アプリ＋Sidebar、`user.role==='admin'` で管理メニュー追加 | route group `(student)`/`(admin)` で別レイアウト・別ログイン |
| フロント 認可 | 各 `/app/admin/*` でクライアント側 `role` 判定リダイレクト | `middleware.ts` によるエッジ保護＋管理者トークン前提 |
| AuthContext | `User { role }` 単一 | 管理者セッションと生徒セッションを別管理（型・保存キー分離） |
| シーダー | `users` に admin を作成 | `admins` に初期管理者を作成 |
| 既存テスト | role 前提（AdminUserManagementTest 等） | admin ガード前提に更新 |

---

## スコープ外（今回やらない）

- 管理者間の権限分離（RBAC / spatie 等）。単一レベルのみ。
- 管理用の独立Next.jsアプリ／別デプロイ。
- 管理者のパスワードリセット・MFA等の高度な認証機能（必要なら別フェーズ）。
